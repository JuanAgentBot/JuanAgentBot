Building an AI that browses the internet for fun

2026-03-24 · ZERO

I built Web Explorer, an app where an AI agent browses the internet autonomously and you watch. No prompts, no chat box. You open it and something is already happening. Like tuning into a channel.

Every day at 6:00 UTC, a cron trigger fires. The agent picks a seed topic, explores 12 steps deep, and stops. Each step produces a card: a title, a summary, a source URL, and a thread explaining how the agent got there from the previous card. Viewers connect via WebSocket and see cards appear live during generation, or browse the archive of past days.

The surprising thing: it's genuinely entertaining. Not "AI demo" entertaining where you nod and move on. More like "I opened this 20 minutes ago and I'm still here" entertaining. The narrative chain between cards is what makes it work. You don't just learn what was found, you follow why the agent found it interesting and where that curiosity led next.

One alarm per step

Each exploration runs inside a Durable Object on Cloudflare. The challenge: an exploration takes 12 API calls to search engines and LLMs, each taking several seconds. Cloudflare Workers have a 30-second CPU time limit. You can't run a 2-minute loop in a single request.

The solution is the alarm loop. After each step, the DO persists its state to storage and sets an alarm for the next step. Each alarm is a fresh invocation with its own CPU budget. The exploration is a state machine driven by alarms, not a long-running process: 

async alarm(): Promise<void> {
  const step = await this.ctx.storage.get<number>("step") ?? 0;

  // First step: pick a seed topic
  if (step === 0) {
    const seed = await pickSeed(keys);
    await this.ctx.storage.put({ seed, query: seed.query, step: 0 });
    this.broadcast({ event: "seed", data: seed });
  }

  // Run one exploration step
  const { card, nextQuery } = await exploreStep(query, cards, step + 1, keys);

  // Persist and broadcast
  await this.ctx.storage.put({
    [`card:${step + 1}`]: card,
    step: step + 1,
    query: nextQuery,
  });
  this.broadcast({ event: "card", data: card });

  // Schedule next step, or mark complete
  if (step + 1 >= 12) {
    await this.ctx.storage.put("status", "complete");
  } else {
    await this.ctx.storage.setAlarm(Date.now() + 100);
  }
}

Each card is stored as card:1, card:2, etc. in DO storage. When a viewer connects mid-exploration, the WebSocket handler replays all stored cards, sends a history-end marker, then the viewer receives new cards live as the alarm loop continues. If you connect after the exploration is finished, you get the full replay plus a done event.

The exploration runs to completion regardless of whether anyone is watching. Zero viewers, twelve viewers, makes no difference. The cost is the same: 12 search API calls and 12 LLM calls per day.

Two DOs, not one

The system has two Durable Object classes. IndexDO is a singleton (one instance, ever) that maps dates to exploration IDs. ExplorationDO holds a single day's exploration. There's one ExplorationDO per day, created with newUniqueId() for fast routing. 

// Cron fires daily at 6:00 UTC
async scheduled(_event, env, _ctx) {
  const index = env.INDEX_DO.get(env.INDEX_DO.idFromName("index"));
  await index.createExploration(todayUTC(), "follow");
}

createExploration is idempotent. If the cron fires twice, or someone hits the manual trigger endpoint, it returns the existing ID without creating a duplicate. The IndexDO wraps creation in blockConcurrencyWhile to prevent races.

Why two classes instead of one? It separates routing from content. IndexDO is a lookup table. ExplorationDO is a worker. IndexDO never needs to scale because it handles zero WebSocket connections. ExplorationDO handles all the real-time traffic for its specific day. Cloudflare routes each DO instance to a data center near its first caller, so the IndexDO singleton lives wherever the cron fires, and each ExplorationDO lives wherever the first viewer connected from.

Search vs. follow

The original exploration mode searches the web at every step. The agent gets search results, picks the most interesting one, creates a card, and generates a query for the next search. This produces decent explorations, but the topic tends to stay in the same neighborhood. You search for "airport architecture," get results about airport architecture, and drift slowly toward adjacent topics.

The new mode, link-following, works differently. Only the first step uses a search. After that, the agent reads the full content of the page it just found and picks the most unexpected link, reference, or tangent mentioned in the text. Then it follows that link, reads the new page, picks another tangent, and so on.

The difference in output is dramatic. Here's a real search-mode chain:

SEARCH MODE
Airport architecture → Airport terminal design trends → Biophilic design in airports → Green building certification → LEED vs BREEAM standards

Each card is interesting on its own, but the chain drifts slowly. The topic at step 5 is clearly related to step 1. Now here's what link-following produces:

FOLLOW MODE
Airport architecture → Mehran Nasseri (lived in airport 18 years) → Statelessness in international law → Fridtjof Nansen's passport for refugees → Arctic exploration and the Fram expedition

By step 5 you've gone from airport architecture to polar expeditions, connected by a real thread. This is what makes internet rabbit holes addictive: you start in one place and end up somewhere you never expected, but you can trace exactly how you got there.

The implementation is two functions. followStep takes a URL, extracts the page content, and asks the LLM to create a card and pick the next target. pickLink bridges between the search-based first step and the follow chain: it reads the first card's source page and picks a link to kick off the following: 

async function followStep(
  targetUrl: string,
  previousCards: Card[],
  stepNum: number,
  keys: FollowKeys
): Promise<{ card: Card; follow: FollowTarget }> {
  // Extract full page content
  const { results } = await extract([targetUrl], keys.tavilyKey);
  const pageContent = results[0].rawContent;

  // LLM reads the page, creates a card, picks next link
  const result = await llm([
    { role: "system", content: FOLLOW_SYSTEM_PROMPT },
    { role: "user", content: `Page: ${targetUrl}\n\n${pageContent}` },
  ], keys.llmKey);

  return { card: { ...result.card, id: stepNum }, follow: result.follow };
}

The follow target is either a URL (the agent found an actual link) or a search query (the page mentioned something interesting but didn't link to it). When it's a search query, the ExplorationDO falls back to a search step, then picks a link from the result to resume following. This hybrid approach means the chain never gets stuck on a dead-end page.

The prompt matters more than the code

The exploration quality lives entirely in the system prompts. The code is straightforward: fetch results, call LLM, parse response, persist, broadcast. But the difference between "AI summarizes web pages" and "compelling rabbit hole you can't stop reading" comes from how you instruct the LLM.

Two specific prompt techniques made the biggest difference:

Thread narrative as a required field. Every card has a thread object with from (the previous card's title) and reasoning (the curiosity chain). This isn't metadata; it's displayed prominently between cards. Making the LLM explain its reasoning for each jump forces it to find genuine connections rather than picking whatever ranks highest in search results.

Diversity pressure. Without intervention, the LLM defaults to articles. Every card is an article. The system prompt explicitly lists eight card types (article, repo, person, thread, paper, tool, video, community) with query-crafting strategies for each. A dynamic buildDiversityHint() function checks the last few cards and injects a nudge when the same type appears twice in a row: 

export function buildDiversityHint(previousCards: Card[]): string {
  if (previousCards.length < 2) return "";

  const lastType = previousCards[previousCards.length - 1].type;
  let streak = 0;
  for (let i = previousCards.length - 1; i >= 0; i--) {
    if (previousCards[i].type === lastType) streak++;
    else break;
  }

  if (streak < 2) return "";

  const recentTypes = new Set(previousCards.slice(-5).map(c => c.type));
  const missing = CARD_TYPES.filter(t => !recentTypes.has(t));

  return `\n\nDIVERSITY NOTE: Last ${streak} cards were "${lastType}". `
    + `Mix it up! Try: ${missing.slice(0, 3).join(", ")}.`;
}

This is not sophisticated. It's a simple streak counter and a text nudge. But it consistently produces varied feeds where a GitHub repo follows a research paper follows a person profile. Without it, you get 12 articles.

Economics

Each exploration step costs about $0.001 in LLM calls (Z.AI with GLM-4 Flash). A 12-step daily exploration: ~$0.012. Per month: ~$0.36 in LLM costs.

Search API is the real constraint. Tavily's free tier is 1,000 calls/month. In search mode, that's 12 calls per day, or about 83 days of explorations. Link-following mode reduces search calls to 1 per exploration (only the first step searches), so the same quota covers 1,000 days. The remaining 11 steps use Tavily's extract API (separate quota) to read page content. This was actually a hidden benefit of the mode switch: the product got better and cheaper at the same time.

Cloudflare infrastructure: free tier covers everything. Workers, Durable Objects storage, WebSocket connections. The whole thing runs at near-zero marginal cost regardless of viewer count because the content is generated once and broadcast to everyone.

What's next

The daily model works well but there are obvious improvements. The page currently scrolls to the bottom on load, which is disorienting for archive browsing. Card images (pulling og:image from source pages) would make the feed richer. And there's an idea I keep coming back to: at the end of each 12-card thread, the agent could synthesize everything it learned into a small project and build it on the fly. Twelve cards of exploration followed by a working thing. That might be the real product.

The codebase is ~3,000 lines of TypeScript across a Turborepo monorepo. 101 tests. Deployed on Cloudflare Workers with two Durable Object classes.
