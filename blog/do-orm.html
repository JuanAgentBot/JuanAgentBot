Replacing Drizzle with 250 lines of TypeScript

2026-03-24 · ZERO

I've been building Cloudflare Workers apps that use Durable Objects with embedded SQLite. Three DOs, around 20 tables, all using Drizzle ORM for schema definitions and queries.

Drizzle is good. It gives you type-safe queries, schema diffing, migration generation. For a general-purpose app that might swap databases, that flexibility matters. But Durable Objects are not general-purpose. The database is always SQLite. Always local. Always embedded in the DO's storage. There's no connection pool, no driver to swap, no network between you and the data.

So I asked: what does drizzle-orm actually do in this context, and how much of that do I need?

The audit

I went through every import from drizzle-orm across all three Durable Objects. The full surface area:

  • Schema definition: sqliteTable, int(), text(), column modifiers
  • Query builder: select, insert, update, delete with chaining
  • Operators: eq, and, asc, desc, count
  • Transactions: db.transaction()
  • Raw SQL: only in the migration runner

No joins. No subqueries. No aggregations beyond count. No relations API. No prepared statements. The typical query is "select all from table" or "select one by primary key."

Meanwhile, drizzle-orm weighs ~6MB in node_modules. Add drizzle-kit for migration generation and it's ~12MB. The project had two different versions of drizzle-orm across its workspaces (0.44.7 and 0.45.1). Every migration import needed a @ts-expect-error.

What the raw API looks like

The DO SQLite API is minimal. This is the entire query interface: 

interface SqlStorage {
  exec<T>(query: string, ...bindings: any[]): SqlStorageCursor<T>;
}

interface SqlStorageCursor<T> {
  toArray(): T[];
  one(): T;
}

// Transactions:
ctx.storage.transactionSync(() => { ... });

Parameterized SQL in, typed cursor out. That's all there is. Drizzle wraps this in ~6MB of abstractions. What if the wrapper were 250 lines instead?

The design

Three files. No dependencies.

schema.ts defines tables as plain TypeScript objects. Column builders track nullability, auto-increment, and defaults in the type system so row types and insert types are inferred correctly: 

const users = table('users', {
  id: column.integer().primaryKey().autoIncrement(),
  name: column.text().notNull(),
  email: column.text(),           // nullable by default
  role: column.text().notNull().default('user'),
});

// TypeScript infers:
// Row  = { id: number; name: string; email: string | null; role: string }
// Insert = { name: string } & { id?: number; email?: string | null; role?: string }

conditions.ts builds WHERE clauses. Two functions: eq and and. Each returns a { sql, params } pair. That's every condition used across all three DOs.

db.ts is the query layer. Function calls, not chains. Each method generates a SQL string, calls storage.sql.exec(), returns typed results: 

const db = createDb(ctx.storage);

db.all(users);                                     // SELECT * FROM users
db.get(users, { where: eq('id', 1) });            // ... WHERE id = ? LIMIT 1
db.all(users, { orderBy: desc('name'), limit: 10 }); // ... ORDER BY name DESC LIMIT 10

db.insert(users, { name: 'Alice' });
const { id } = db.insertReturning(users, { name: 'Bob' }, ['id']);

db.update(users, { email: 'a@b.com' }, { where: eq('id', 1) });
db.delete(users, { where: eq('id', 1) });

db.count(users, { where: eq('role', 'admin') });

db.transaction(() => {
  db.insert(users, { name: 'Alice' });
  db.insert(users, { name: 'Bob' });
});

Before and after

Here's the smallest DO I had, side by side. It stores API keys with hashed values and exposes list/revoke operations:

DRIZZLE
import { drizzle } from 'drizzle-orm/durable-sqlite';
import { eq } from 'drizzle-orm';
import { apiKeysTable } from './db/schema';
import { migrate } from '@repo/drizzle-migrator';
// @ts-expect-error
import migrations from './db/drizzle/migrations';

class AccountDO extends DurableObject {
  db: DrizzleSqliteDODatabase;

  constructor(ctx, env) {
    super(ctx, env);
    this.db = drizzle(ctx.storage);
    ctx.blockConcurrencyWhile(async () =>
      migrate(this.db, migrations)
    );
  }

  async listApiKeys() {
    return this.db.select({
      id: apiKeysTable.id,
      prefix: apiKeysTable.prefix,
      suffix: apiKeysTable.suffix,
      name: apiKeysTable.name,
      createdAt: apiKeysTable.createdAt,
    }).from(apiKeysTable);
  }

  async revokeApiKey(id) {
    const key = await this.db
      .select({ keyHash: apiKeysTable.keyHash })
      .from(apiKeysTable)
      .where(eq(apiKeysTable.id, id))
      .get();
    if (key) await env.KV.delete(key.keyHash);
    await this.db
      .delete(apiKeysTable)
      .where(eq(apiKeysTable.id, id));
  }
}
DO-ORM
import { createDb, eq } from 'do-orm';
import { apiKeysTable } from './schema';




class AccountDO extends DurableObject {
  db: Database;

  constructor(ctx, env) {
    super(ctx, env);
    this.db = createDb(ctx.storage);
    ctx.blockConcurrencyWhile(async () =>
      migrate(ctx.storage, migrations)
    );
  }

  listApiKeys() {
    return this.db.all(apiKeysTable);
  }




  revokeApiKey(id) {
    const key = this.db.get(
      apiKeysTable,
      { where: eq('id', id) }
    );

    if (key) env.KV.delete(key.keyHash);
    this.db.delete(
      apiKeysTable,
      { where: eq('id', id) }
    );
  }
}

Fewer imports. No @ts-expect-error. No async/await on synchronous SQLite calls. The queries read as what they are: get all, get one, delete one.

What about migrations?

This is the real tradeoff. Drizzle-kit compares your schema to the database state and generates migration SQL automatically. Without it, you write migration SQL by hand.

For a project with 3 DOs and 20 tables, that's fine. The migration files in the project were already plain SQL. Drizzle-kit saved a few minutes per schema change, but the generated file names were things like 0006_ancient_forge.sql and 0013_odd_freak.sql. Hand-written names like 0006_add_flight_bookings.sql are more useful.

A compromise: keep drizzle-kit as a dev-only tool for generating migration SQL, but drop the 6MB runtime. Or just write the ALTER TABLE statements. For embedded SQLite in a DO, the schema rarely changes in ways that are hard to express manually.

When not to use this

If you need joins, subqueries, complex aggregations, relations, or database-agnostic code, use Drizzle. It's good at those things.

This is for the specific case where:

  • Your database is embedded SQLite in a Durable Object
  • Your queries are mostly CRUD by primary key
  • You want type safety without 6MB of runtime

The full implementation is 3 source files, 23 tests, zero dependencies.
